# Visual Direction v0

Status: Approved product direction
Purpose: Defines Workprint's visual and experiential philosophy before
implementation
Scope: Product feel, interaction posture, trust presentation, and design
principles for the first web experience
Preferred direction: Warm Investigator

This document builds on the approved web experience product contract in
`docs/web-experience-v0.md`. It defines how Workprint should feel before
implementation choices are made. It does not choose a frontend framework,
component library, visual system, colors, typography, or production UI
components.

Benchmark question:

"If Apple, Notion, Linear, and Arc all built this product, what would
Workprint deliberately do differently?"

Workprint should borrow their care for clarity, restraint, craft, and flow,
but it should not become a lifestyle object, a workspace dashboard, an issue
tracker, or a browser-like productivity surface. Workprint's difference is
warm, evidence-bound investigation made calm enough for a novice to trust.

The preferred visual direction is **Warm Investigator**.

Internal emotional promise:

"Your work deserves to be seen clearly."

Internal creative principle:

"Recognition before analysis."

## Product Personality

Workprint is an investigative companion rather than a reporting tool.

The preferred direction is Warm Investigator: personal enough to recognize the
user's work, precise enough to preserve evidence boundaries, and restrained
enough that trust comes from traceability rather than visual persuasion.

It should feel:

- calm;
- warm;
- personal;
- reflective;
- credible;
- evidence-driven;
- quietly encouraging.

It should avoid:

- AI hype;
- dashboards for the sake of dashboards;
- corporate polish;
- clinical detachment;
- futuristic spectacle;
- enterprise complexity;
- unnecessary animations;
- visual clutter;
- gamification;
- overly sentimental language or imagery;
- cuteness;
- decoration that does not help understanding;
- generic AI dashboard patterns;
- anthropomorphic AI.

The companion quality should come from pacing, language, and evidence-aware
guidance, not from pretending the product is a person. Workprint should feel
like a careful investigator sitting beside the user, organizing what can be
known and naming what cannot.

## Brand Attributes

Workprint should be:

- Precise without feeling cold.
- Warm without weakening precision.
- Minimal without becoming empty.
- Serious without becoming corporate.
- Helpful without becoming chatty.
- Confident without overclaiming.
- Modern without becoming futuristic.
- Personal without becoming anthropomorphic.

Its brand center is not "AI-powered reports." Its brand center is
"traceable understanding."

## Emotional Journey

The first web experience should move the user through four emotional states:

| Screen | User feeling | Product response |
| --- | --- | --- |
| Start | "I have a project, but the history is scattered." | Offer one clear prompt and one next action. |
| Add where the work happened | "I know where things happened, but not what matters." | Accept files and folders calmly; explain source limits locally. |
| Investigating | "Is this doing something trustworthy?" | Show simple progress and reassure that Workprint is organizing evidence, not guessing. |
| Discoveries | "What did it find, and can I believe it?" | Lead with one supported insight, then show why Workprint believes it. |

The journey should feel like clarity emerging, not like a workflow being
completed.

## Design Principles

- Recognition before analysis. The product should first help the user feel
  that their work has been seen clearly, then explain the evidence underneath.
- Warmth must never weaken precision. Warm copy, spacing, and tone are welcome
  only when they preserve attribution limits, source boundaries, and unknowns.
- The user's work is the visual focus. The interface should not make the
  Workprint system, AI, charts, controls, or brand decoration the star.
- Evidence should feel close and understandable. Support should appear like
  annotations or thoughtful notes near the claim it explains.
- First insights need generous space. The first discovery should have room to
  land before the report, export controls, or secondary sections appear.
- Unknowns should feel honest, not disappointing. Limits should be calm,
  visible, and matter-of-fact.
- Trust comes from restraint and traceability. Workprint should persuade by
  showing its work, not by visual drama.
- Avoid wrapping every section in cards. Containers should create structure
  only when structure is needed.
- Use progressive disclosure instead of visual density. Detail should unfold
  when it helps the user inspect, decide, or export.
- Personal does not mean anthropomorphic. Workprint can feel attentive without
  becoming a character, chatbot, or simulated colleague.
- Start with one question. The first screen should feel closer to a search box
  than a setup form.
- Let the product grow only after the user acts. Complexity should unfold from
  the user's own project history.
- Make support visible before detail. A claim should be followed by "why
  Workprint believes this" before the user is asked to read a report.
- Treat unknowns as part of quality. Unknowns should look intentional, not
  like errors or missing product work.
- Use quiet structure. Layout should guide the eye without turning every
  section into a card or dashboard tile.
- Keep the report secondary to the first insight. The first discovery is the
  moment of value; the report is the expandable account behind it.
- Prefer language over decoration. If a visual element does not improve
  comprehension, confidence, or navigation, it should not be there.

## Interaction Principles

- One obvious next action per screen.
- Supporting details should be available, not forced.
- Trust disclosures should sit beside the action they affect.
- Evidence drill-down should feel like opening the reasoning behind a claim,
  not switching to an audit console.
- Backtracking should preserve user work.
- The product should not punish uncertainty. If the evidence is limited, the
  next action should help the user understand the limit or add more history.
- Export should feel like preserving the account, not publishing a polished
  claim.

## Information Hierarchy

Workprint should order information by the user's question:

1. What am I doing here?
2. What does Workprint need from me?
3. What is happening now?
4. What did Workprint find?
5. Why should I believe it?
6. What can it not determine?
7. What can I inspect or export?

The first supported insight should dominate the first Discoveries viewport.
The report preview, section navigation, export controls, and secondary
details should appear after a clear visual break or deliberate continuation.

The user's work should be the visual subject. Interface elements should frame
the work, not compete with it. The hierarchy should make the claim, support,
unknown, and evidence path easy to follow before asking the user to interpret
sections, filters, metadata, or exports.

## Motion Philosophy

Motion should clarify state changes, not perform intelligence.

Use motion to:

- reveal progressive detail;
- show that investigation is moving through calm stages;
- open and close evidence drill-down without disorienting the user;
- mark the transition from waiting to first discovery.

Avoid motion that:

- implies the product is thinking like a person;
- makes evidence feel magical;
- draws attention away from the claim and support;
- creates urgency, celebration, or gamified reward;
- delays the user's ability to inspect evidence.

Motion should feel patient and reversible. It should never be required to
understand the interface.

## Typography Philosophy

Typography should be editorial and highly readable. It should communicate
hierarchy, calm, and evidence boundaries without relying on font personality
or naming a specific font.

The typographic system should:

- make the primary prompt and first insight immediately legible;
- distinguish claim, support, confidence, and unknown;
- keep long report sections readable;
- make source labels and evidence references scannable;
- avoid oversized marketing-style text once the user is inside the product;
- avoid tiny legal-style trust disclosures.

The voice of type should be editorial and investigative rather than
decorative, technical, or sales-led.

## Color Philosophy

Color should be warm but specific, neutral but not generic, and quiet enough
that the user's work remains the focus. This document does not choose a
palette or color values.

Color should:

- support calm focus;
- distinguish ready, limited, needs-action, and unknown states;
- make trust and confidence visible without implying numerical precision;
- reserve stronger emphasis for first insight, action, and genuine warnings;
- keep evidence drill-down readable and grounded.

Color should not:

- create a one-note palette;
- use hype-coded AI gradients as the main identity;
- replace generic AI-purple with generic cream, serif, and sage styling;
- make uncertainty look dangerous by default;
- use red for every limitation;
- rely on color alone to communicate status.

The visual direction should avoid both common defaults: the purple-blue AI
glow and the tasteful cream editorial template. Warm Investigator needs a
reasoned visual language, not a borrowed aesthetic costume.

## Layout And Spacing Philosophy

Spacing should be generous enough for reflection. Workprint should give the
first insight, supporting notes, and unknowns room to breathe.

Use spacing to:

- separate recognition from analysis;
- protect the first insight from report density;
- make evidence annotations feel close to the claim;
- reduce anxiety during upload, investigation, and export;
- create calm rhythm between prompt, action, support, and detail.

Containers should be restrained. They should group repeated items, evidence
details, or focused actions, but Workprint should avoid wrapping every section
in a card. Over-containerized layouts make the product feel like a generic
dashboard and weaken the sense that the user's work is the subject.

Evidence should often feel like annotations, marginal notes, or supporting
context near a claim rather than separate report furniture. The user should
not have to jump across the page to understand why Workprint believes
something.

## Trust Signals

Trust should appear through repeated small signals:

- claims are followed by support;
- limits are visible near the claim they qualify;
- source status says "ready" or "limited" without implying completeness;
- export warns about excerpts and metadata before the user acts;
- evidence drill-down shows what the evidence does and does not prove;
- contribution percentages never appear;
- unknowns are presented as honest results.

Trust should feel built into the product, not bolted on as a legal notice.
Restraint is itself a trust signal: fewer claims, clearer support, visible
unknowns, and no decorative certainty.

## Confidence Communication

Confidence should be expressed as plain-language bands with short reasons.

The visual treatment should:

- place confidence near the claim it qualifies;
- pair confidence with the evidence reason and the important unknown;
- distinguish confidence from certainty;
- avoid numeric scores unless a future approved milestone introduces them;
- avoid progress-bar metaphors for truth.

Confidence should answer: "Why should I believe this much, and no more?"

## Progressive Disclosure Philosophy

Progressive disclosure is Workprint's main tool for keeping complexity out of
the first experience.

Default view:

- one prompt;
- one upload area;
- one progress story;
- one first insight.

Expanded view:

- source limits;
- evidence references;
- report sections;
- export contents;
- raw metadata only when useful.

Progressive detail should never hide a material limitation. It should hide
complexity until the user needs it, while keeping trust-critical boundaries
near the relevant action or claim.

## Empty States

Empty states should be brief, specific, and useful.

They should:

- name the missing thing in novice language;
- offer one next action;
- avoid blame;
- avoid generic illustrations that do not explain the task;
- make unknown or limited outcomes feel valid when evidence is insufficient.

Examples:

- "Tell Workprint what you are working on."
- "Add conversations, documents, designs, or repository history to begin."
- "Workprint found evidence, but not enough for a reliable first insight yet."

## Loading Experience

Loading should feel like careful reading, not a black box. It should be calm
and quietly encouraging, but never theatrical.

Use the approved stages:

1. Reading your project.
2. Finding important moments.
3. Preparing your discoveries.

The loading experience should avoid logs, model language, complex pipelines,
or theatrical "AI is thinking" cues. It should make the user comfortable
waiting because the product is doing bounded, understandable work.

## Error Experience

Errors should preserve confidence and momentum.

They should:

- say what happened in plain language;
- identify the affected file or source when useful;
- preserve everything else the user added;
- offer retry, remove, replace, or continue when safe;
- distinguish unsupported input from failed investigation;
- avoid implying the user's project is bad or unusable.

Error tone should be steady. The product should sound like it knows how to
recover.

## Discovery Reveal Philosophy

The first discovery is the product's first proof of value.

It should reveal in this order:

1. The claim.
2. Why Workprint believes this.
3. What Workprint cannot determine.
4. The option to inspect evidence.
5. The option to continue to the full report.

The first discovery should not share the viewport with a dashboard grid,
metric row, chart wall, export panel, or report table. It should have enough
space to feel considered and important. The reveal should feel earned rather
than theatrical: the product has read the project history, found something
supported, and now shows the user why it can say it.

## Voice And Tone

Workprint's voice should be plain, thoughtful, and non-technical. It should
sound like careful recognition, not automated judgment.

Use:

- direct sentences;
- human-scale language;
- precise evidence explanations;
- calm acknowledgements of limits;
- active verbs tied to evidenced actions.

Avoid:

- model-flavored labels;
- detection language;
- hype;
- surveillance language;
- generic AI assistant phrasing;
- sentimental praise.

Example tone:

Instead of:

"High-confidence human-led activity detected."

Prefer:

"You repeatedly set the direction."

Then explain the evidence underneath in precise language, such as:

"Workprint found multiple project moments where your messages set constraints,
requested revisions, or chose the next step. This supports direction-setting
within the provided evidence, but it does not prove total authorship or
measure contribution."

## Accessibility Principles

Accessibility is part of trust.

The visual direction should support:

- clear reading order;
- keyboard access for every action and disclosure;
- visible focus states;
- text that remains legible at larger sizes;
- status communication that does not rely on color alone;
- motion that can be reduced without losing meaning;
- evidence drill-down that is understandable to screen readers;
- error messages that are associated with the relevant field or file.

Workprint should not make evidence harder to inspect for users who rely on
assistive technology.

## Design Inspirations

Apple: Learn from restraint, confidence, and careful pacing. Do differently:
avoid product mystique. Workprint must show its evidence instead of asking
the user to trust polish.

Notion: Learn from progressive surfaces and approachable structure. Do
differently: avoid turning Workprint into a workspace of blocks, templates,
and panels before the user has seen a discovery.

Linear: Learn from speed, clarity, and disciplined hierarchy. Do differently:
avoid issue-tracker density and productivity aesthetics that make the
experience feel like work management.

Arc: Learn from calm modernity and flexible interaction. Do differently:
avoid making the product feel like a clever shell around many tools.
Workprint should feel like one focused investigation.

## Explicit Anti-Patterns

Workprint should never feel like:

- an AI hype demo;
- a contribution scoreboard;
- a productivity dashboard;
- an enterprise compliance portal;
- a corporate reporting suite;
- a clinical audit system;
- a futuristic AI cockpit;
- a BI analytics tool;
- a chatty AI assistant pretending to be a person;
- a gamified achievement system;
- an overly sentimental scrapbook;
- a cute productivity toy;
- a decorative editorial template;
- a file parser with a nicer skin;
- a wizard that stops users on explanatory pages;
- a design system showcase;
- a legal disclaimer wrapped in a product;
- a black box that produces confident prose without support;
- a generic AI dashboard with Workprint copy pasted in.

## Unslop UI Guardrails

Future Workprint web interfaces must keep the existing `unslop-ui` guardrail
requirements from `docs/executive-report-design.md`. Before implementation,
contributors must use the upstream guidance and scanner, choose a concrete
visual reference or named design direction, define color and typography
choices with reasons, base layout on the task rather than a generic SaaS
template, and document major design decisions.

Warm Investigator is the named direction for this milestone. It is not
permission to fall into another generic default. Workprint must not replace
generic AI-purple design with generic cream, serif, and sage styling. It must
also avoid untouched framework defaults, gradient heading text, decorative
glow, emoji-as-icons, centered hero plus three feature cards,
interchangeable dashboard card grids, and aesthetic choices that do not serve
evidence clarity.

Never replace one generic visual default with another. Preserve
accessibility, evidence clarity, and usability over aesthetic novelty.

## Open Design Questions

- What visual treatment best distinguishes claim, support, confidence, and
  unknown without making the screen feel analytical?
- How much of the report preview should be visible below the first discovery?
- Should evidence drill-down feel more like a side drawer, an inline expansion,
  or a focused overlay?
- How should the product distinguish "limited" from "needs action" without
  making limitations feel like errors?
- What tone should empty states use when Workprint cannot produce a first
  insight?
- How should export feel quiet and trustworthy without becoming hidden?
- What concrete visual references can support Warm Investigator without
  becoming corporate, clinical, cute, or generically editorial?
